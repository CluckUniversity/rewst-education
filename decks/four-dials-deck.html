02 / 06
The framework

Four independent questions

Capability → the model

Did this need top-tier reasoning, or would a faster, cheaper model have handled it fine?

Effort → the dial

Within that model, did it cut corners it shouldn't have, or was the depth already enough?

Process → a skill

Is the right way to do this worth writing down and reusing, or are you still figuring it out as you go?

Autonomy → an agent

Did this need multiple steps done without checking in, or is one clean exchange enough?

The mistake: collapsing four dials into one decision.

03 / 06
Model vs effort

Model is the ceiling.
Effort is the fill.

WHAT THE TASK NEEDS
Smaller modelmax effort
CEILING
Bigger modellow effort
If it lacked knowledge or capability Escalate the model
If it didn't try hard enough on a clear task Raise the effort

A smaller model at max effort can beat a bigger model at low effort — until the task needs a ceiling the smaller model doesn't have.

04 / 06
Strategy + skills

Where to start, and when to write it down

Step down Start at the most capable model. Step down until quality drops. Repeated task with a fixed, measurable bar — like coding.
Escalate Start where you'd normally land. Escalate only if it falls short. One-off, ad hoc requests.

The decider isn't technical vs non-technical. It's repeats with a measurable bar vs a single ask.

Frequency ≠ skill-worthy
Frequent, but the process is still shifting Not ready to encode
Rare, but high stakes to get right Worth encoding

Author at max, run lean. Build and validate the skill with the most capable model at high effort — every future run inherits that judgment. Once it's solid, the runs can drop down.
